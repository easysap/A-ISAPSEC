§ 01

Architecture mindset

The core philosophy here is quite simple: SAP is the source of truth, but not the place for heavy analytics. Trying to run intensive machine learning inside the ABAP stack is typically a bad idea for performance reasons. Instead, we extract that data to a specialized environment—a data lake or a cloud ML platform—and only send the high-value risk scores back into the security ecosystem.

In my view, the benefit of this modular approach is that it respects the stability of the SAP core while allowing the security team to use modern tools like Python or Spark. This is how you reduce the MTTD without making the Basis team nervous about system resources.

§ 02

Where logs live

Finding the right signal is usually half the battle. In the SAP universe, we look at four key areas that provide the richest context for security models.

User behavior (STAD)

The workload tables, STAD and ST03N, are essentially the fingerprints of activity. They record what T-codes were used and by whom. Establishing a "normal" baseline here is what makes anomaly detection actually work.

Behavioral LogsTransaction History

The Change Audit

Tables like CDHDR and CDPOS track modifications to master data. If someone changes a vendor's bank details without a proper ticket, these tables are the only place that evidence lives. For fraud detection, these are non-negotiable.

Change DocumentsMaster Data Integrity

Access Metadata

The USR* and AGR* family of tables tell the story of "who can do what." We pull these to understand the authorization graph, which is vital for detecting if a user suddenly gains permissions they shouldn't have.

Roles & ProfilesSoD Analysis

The Security Log (SM20)

This is the classic audit log. It captures failed login attempts and sensitive RFC calls. One must remember to verify that the audit classes are actually turned on in SM19, otherwise, the model will be looking at an empty table.

§ 03

Pulling the data

Extraction isn't just about moving data; it's about doing it without breaking the system. Depending on the landscape, you might choose a slow-and-steady approach or a real-time stream.

For master data, a daily batch is usually fine. But for the Security Audit Log, we typically want something faster. Many modern architectures are moving toward OData services because they are easier for cloud-native tools to consume compared to traditional RFC calls.

§ 04

The feature layer

Raw SAP tables are "dirty" from a data science perspective. You can't just throw a raw USR02 table into a model and expect magic. We have to engineer features that represent risk. For example, instead of just a timestamp, we create a feature called "Activity Outside Business Hours" or "Deviation from Peer Group Median."

A Professional Tip

Pay close attention to timezone normalization. SAP systems often run on UTC, while users are across the globe. If you get this wrong, your "late night login" alerts will be full of false positives from users in different regions.

§ 05

Model selection

Not every problem needs a Neural Network. In fact, for most SAP security use cases, simpler models are often more explainable—and in a regulated audit environment, explainability is a requirement.

Category Typical Algo Use Case
Unsupervised Isolation Forest Detecting rare transaction jumps
Supervised XGBoost Predicting high-risk role changes
Graph Analysis Node2Vec Identifying hidden SoD paths
§ 06

SIEM workflows

The AI model's job isn't to fix the problem; it's to provide the SOC analyst with a "Reason for Suspicion." We integrate this by sending a JSON payload to the SIEM—like Sentinel or Splunk—containing the User ID, the Risk Score, and a short summary of the anomaly.

§ 07

Response strategies

When the AI detects something, the response needs to be proportional. We categorize these into two buckets: actions that happen automatically and those that need a human eyes-on check.

Human Intervention

  • Manager review of activity
  • Manual account lock for VIPs
  • Detailed forensic deep-dive

Automated "Guardrails"

  • Revoking "Emergency" access
  • Triggering MFA re-auth
  • Isolating specific RFC users
§ 08

Closing thoughts

Securing an SAP landscape with AI is a journey of data quality. It is better to have one reliable model running on a nightly batch than a complex real-time system that produces too much noise. Start with the most critical logs, build trust with the Basis team, and expand the architecture as the data matures.
