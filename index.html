§ 01

Architecture overview

SAP environments produce a surprising amount of security-relevant data. Role assignments in USR*, audit events in SM20, transaction records in STAD — it's all there. Most organisations just aren't doing much with it. The logs age off, the role tables sit untouched, and the first sign of a problem is often a fraud investigation six months after the fact.

The architecture here is built around one rule: SAP is the data source, not the analysis platform. AI runs externally — in your data lake, your cloud environment, wherever you do ML — and the results go back into your SIEM. Nothing gets deployed into the SAP kernel. The production system stays clean.

Each layer can be swapped out independently. Change from Splunk to Sentinel, or switch from a nightly ETL batch to Kafka streaming — the AI models don't care. The contracts between layers are simple: REST/OData out of SAP, feature vectors into the models, JSON risk scores into the SIEM. That's intentional. Don't let the integration complexity creep into the intelligence layer.

§ 02

SAP data sources

Four table families inside SAP carry the signals that matter. Before you design anything, it's worth understanding what each one actually captures — and where it falls short.

User activity logs

Workload Monitor tables STAD and ST03N record every dialog transaction a user runs: program name, T-code, runtime, CPU usage, step count. This is your main source for behavioural baselining — what does normal look like for this user, this role, this time of day.

One thing to watch: STAD retention defaults to 2–7 days in most systems. If you're not streaming or extracting it regularly, the data is gone.

STADST03Nbehavioural baseline

Change document logs

Tables CDHDR (header) and CDPOS (line items) track field-level changes to master data — vendor bank accounts, customer records, cost centre assignments, config settings. Who changed what, when, and from which value to which value.

In my experience, this is where the most interesting fraud signals live. A changed vendor IBAN that nobody flagged is a classic pattern.

CDHDRCDPOSfraud detection

User & role tables

The USR* family covers user master records, logon data, and parameter assignments. AGR_* holds role definitions and authorization objects. Together they give you the full access graph — who has what, and how they got it.

This is your input for SoD analysis and graph-based privilege escalation detection. It's also the part that tends to be messiest in long-running SAP landscapes.

Security audit log

Transaction SM20 reads the Security Audit Log — failed logins, RFC calls, report executions, user locks and unlocks. It has to be switched on per client in SM19 with the right audit class filters. A lot of systems have it partially configured or off entirely. Worth verifying your own landscape before assuming the data is there.

Architecture note

Don't pull these tables directly via SELECT * using a highly privileged RFC user. Use purpose-built extraction function modules or OData services with minimum-privilege service accounts. Direct table access skips SAP's authorisation checks at the application layer — which is exactly the kind of thing this whole pipeline is meant to detect.

§ 03

Extraction methods

There's no single right answer here — it depends on how much lag you can tolerate and what infrastructure you already have. Most real-world setups run two tracks: a nightly batch for bulk historical data, and a streaming path for the stuff you actually need in real time.

OData / REST APIs

SAP Gateway exposes OData v2/v4 services for structured reads of user, role, and master data. S/4HANA ships many of these out-of-the-box via the API Business Hub. Good fit for scheduled pulls of slowly-changing data — role assignments, user master records — where a few hours of lag is fine.

SAP GatewayBTPOData v4

RFC connectors

RFC calls invoke ABAP function modules directly over SAP's proprietary protocol. Libraries like pyrfc (Python), SAP JCo (Java), or SAP NCo (.NET) handle the transport layer. Useful when you need on-demand extraction or want to call a custom ABAP extractor that joins across several tables at once.

pyrfcSAP JCoSAP NCo

ETL tools

Informatica IDMC and SAP Data Services both ship certified SAP adapters built on top of the RFC layer, but they add scheduling, error handling, lineage tracking, and transformation logic on top. If you're doing bulk historical loads into a data lake, this is the path of least resistance.

Informatica IDMCSAP DS

Log streaming

SAP Event Mesh on BTP, or a local log-forwarding agent, can publish SM20 audit events to a Kafka topic as they happen. Your AI scoring service consumes the topic and writes risk scores back to the SIEM within seconds. This is the architecture that gets you sub-minute detection — but it takes real infrastructure investment to run reliably.

§ 04

Data processing & feature engineering

Raw SAP data won't go straight into a model. You need a processing layer that turns it into feature vectors — and that also deals with the messy reality of SAP extracts: duplicate records from change document writes, timezone mismatches across system landscapes, sparse categorical fields in role tables. This work is unglamorous but it's where most projects quietly fail.

Key engineered features

Temporal features

Transaction count per hour/day, time-of-day buckets, deviation from the user's historical activity window (e.g. first login at 03:00 when baseline is 08:00–18:00), weekend activity flags.

Access features

Sensitive T-code execution frequency (SE16, SU01, FB60), number of distinct authorization objects accessed per session, SoD conflict score derived from role intersection analysis.

Volume features

Data records read/written per session, bulk download indicators (high record count in single transaction), comparison against peer group median (same role, same department).

Graph features

Role inheritance depth, number of directly assigned vs. inherited authorisations, centrality score in the user-role-object graph — inputs for privilege escalation path detection.

Engineering tip

Version your feature store alongside your models, and treat feature drift as a first-class concern. SAP landscapes change — new roles get added, T-codes get deprecated, custom Z-transactions come and go. A feature that was stable for a year can become noise after an upgrade. I've seen models degrade silently for weeks before anyone noticed the inputs had shifted.

§ 05

AI model layer

Four model families cover the main threat categories. In practice, you don't route each model's output to the SIEM independently — that's a fast way to drown your SOC in alerts. Combine them into a unified risk score first, then forward that. The table below covers what each model type does and what it needs to get started.

Model type Algorithms Detects Training
Anomaly detection Isolation Forest, Autoencoders, OCSVM Unusual transaction patterns, off-hours access, data volume spikes Unsupervised — no labelled data required
Risk prediction Random Forest, XGBoost, LightGBM High-risk user sessions, policy violations, SoD conflicts in use Supervised — requires labelled historical incidents
Sequence models LSTM, Transformer (fine-tuned) Multi-step attack chains, kill-chain progression, session hijacking Semi-supervised — sequence of events as input
Graph models GNN, PageRank variants, path analysis Privilege escalation paths, role mining anomalies, dormant account abuse Unsupervised / rule-augmented

Start with anomaly detection — no labelled data required, and you'll have findings within weeks. The supervised risk model comes later, once your security team has built up 6–12 months of investigated incidents to train on. Jumping straight to supervised learning without labelled data is one of the most common mistakes I see on these projects.

§ 06

SIEM integration

The AI layer shouldn't write raw log data back to the SIEM — it should write scored, enriched events. Each one carries a risk score (0–100), a category label, the features that drove the score, and a reference to the originating SAP session or user. A SOC analyst should be able to look at the alert and know what happened without having to open a separate SAP GUI session.

Splunk

The HTTP Event Collector (HEC) is the fastest way to get AI-scored events into Splunk. Write SPL correlation searches that join AI risk scores against your existing SAP add-on data. If you haven't already built a log ingestion pipeline, the SAP Basis add-on covers the basics and saves you a few weeks of plumbing work.

HECSPL correlationSAP add-on

Microsoft Sentinel

SAP maintains a certified Sentinel solution on the Content Hub — it covers SM20, CDHDR, and user master data out of the box. Add custom KQL analytic rules that reference your AI risk score field. For automated response, Logic Apps integrates natively and doesn't require a separate SOAR platform.

Content HubKQL rulesLogic Apps
Integration pattern

Push AI scores as enrichment fields on your existing SAP log events, not as separate event types. If you create a new event type for every AI output, you'll end up with correlation rules that need joins at query time and a schema that nobody can maintain six months later. Keep the structure flat — one event, all the context on it.

§ 07

Action & response layer

This is where most SAP security programs get stuck. You've built the detection. You have a risk score. And then nothing happens for four hours because the alert is sitting in a queue waiting for a human to look at it. By then the session is long over. Design response workflows before you go live, not after the first real incident.

Manual response

  • Security investigation triggered in GRC
  • Access review pushed to user's manager
  • Audit report generated for compliance record
  • Session reconstruction from STAD data
  • Escalation to SAP Basis for technical remediation

Automated response

  • Account locked via SUIM/SU01 RFC call
  • Sensitive role removed from user assignment
  • Approval workflow triggered in SAP GRC Access Control
  • Alert pushed to security on-call channel
  • Correlated ticket auto-created in ITSM platform

A useful rule of thumb: automate the high-confidence, low-blast-radius actions first. Locking a dormant account that activates at 02:00 from an unfamiliar IP is a safe candidate. Pulling a role from an active user who's mid-session is not — that one needs a human in the loop, every time.

§ 08

Batch vs. streaming architecture

People ask which one to build as if it's a choice. It usually isn't — you'll end up running both. The real question is which one to start with, and for what.

Batch processing (most common)

A nightly or hourly ETL job pulls the prior period's STAD, CDHDR, and role data, builds features, runs the models, and loads scores back into the SIEM. Simple to operate. Easy to debug when something breaks. Sufficient for SoD conflict analysis, access certification campaigns, and compliance reporting.

The downside is obvious: if something happens at 09:00, you might not see it until 06:00 the following morning.

Streaming (advanced)

SM20 events forward to a Kafka topic via SAP Event Mesh or a log agent. A stream processor — Flink, Spark Streaming — runs lightweight anomaly detection and writes scored events back to the SIEM within seconds. Sub-minute detection latency is genuinely achievable here.

It's also a lot more infrastructure to run. Schema evolution, stateful processing, consumer lag monitoring — these aren't problems the batch path has. Go in with clear eyes.

Where to start

Build the batch pipeline first. It handles 80% of the security value for a fraction of the operational overhead. And when you're ready to add streaming, you don't rebuild — you reuse the same feature definitions and model artifacts. You're just changing how the data gets there.
